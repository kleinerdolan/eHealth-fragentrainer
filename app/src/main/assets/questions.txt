	1.1 Wie unterteilt sich E-Health?
	{
		- eCare
		- eAdminstration
		- ePrevention
		- eLearning
		- eResearch
		- ePublicHealth
	}

	1.2 Nenne 5 Nutzen/Ziele von E-Health?
	{
		- Dem Patienten den Zugang zu geeigneten medizinischen Maßnahmen eröffnen
		- Bessere medizinische Versorgung durch Behebung von Informationsdefiziten
		- Stärkung der Zusammenarbeit zwischen Ärzten und anderen Gesundheitsberufen, die gemeinschaftlich für die Versorgung eines Patienten verantwortlich sind
		- Patient Empowerment“: dem Patienten mehr Informationen und Verantwortung geben und ihn besser und aktiver in seine Betreuung und Therapie einbeziehen
		- Effektivere, bessere Versorgung durch neue Versorgungsformen
		- Medizinische Informationen für Patienten, Selbsthilfegruppen
		- Mehr Patienten können behandelt werden
		- Direkte Konsultation eines Arztes über das Internet
	}

	1.3 Was ist eAdministration?
	{
		- Administrative Aufgaben bezüglich aller digitalen administrativen Daten wie eÜberweisungen, eQuittungen, eRezept
		- Folie 18
	}

	1.4 Gehört die automatische Auswertung von Röntgenaufnahmen zum Aufgabenbereich von eHealth? Begründen Sie ihre Antwort?
	{
		- Nein. Da:
			* Bei E-Health liegt der Fokus in Anwendungen der med. Informatik in der unmittelbaren oder mittelbaren Patientenversorgung
			* Sowie im Aspekt der Vernetzung von verschiedenen Einrichtungen im Gesundheitswesen
		- Folie 8
	}

	1.5 Was ist Pseudonymisierung und wofür wird es im Kontext von eHealth verwendet?
	{
		- Pseudonym lässt keine Rückschlüsse auf Echte Identität der Person
		- Pseudonym kann aber wiedererkannt werden
		  -> Also nicht anonym
		- (Langzeit-)Studien von Patientendaten
	}

	1.6 Wieso können durch eCare die Kosten nicht groß gesenkt werden?
	{
		- IT Spezialisten und Programmierer kosten zusätzlich Geld
		- Anschaffung, Update und Wartung von IT Systemen
	}

	1.7 Wie kann durch eCare eine Kostensenkung erzielt werden? (2 Möglichkeiten)
	{
		- Vermeidung doppelter Untersuchungen
		- Zeitgewinn durch schnellere Information des Arztes
		- Effektivität der Versorgung steigern, Kostensenkung
	}

	1.8 Erläutern Sie den Begriff „Patient Empowerment! und nennen sie 2 Vorteile?
	{
		Patienten mehr Infos und Verantwortung geben und ihn besser in die Behandlung einbeziehen. Evtl. Beisteuerung von Daten durch Patienten selbst.

		Vorteile:
			* Mehr Daten für den Arzt durch zB Diabetes-Tagebücher
			* Selbstbestimmung über Daten also Kontrolle über Wissen der Ärzte im Bezug auf persönliche Daten
	}

	1.9 Nennen Sie 5 Ziele von eCare?
	{
		- Zugang zu geeigneten medizinischen Maßnahmen dem Patienten eröffnen, wo diese nicht verfügbar sind (Telemedizin, Telemonitoring)
		- Informationsdefizite beheben
		- Kooperative Behandlung erleichtern (Interoperabilität)
		- Patient in Behandlung einbinden
		- Effektivität der Versorgung steigern
	}

	1.10 Wie kann der Datenschutz bei der Weitergabe von Patientendaten insbesondere in Bezug auf klinische Studien geschützt werden?
	{
		- Anonymisierung: Es werden keine persönlichen Informationen bezüglich des Patienten angegeben, außer den benötigten Daten, z. CT-Scan
		- Pseudonymisierung: Es ist möglich, den Patienten zu identifizieren, jedoch durch einen Spitznamen, z. beim Testen neuer Medikamente.
	}

	2.1 Warum sind landesweite E-Health Infrastrukturen sinnvoll?
	{
		- Zur Gewährleistung landesweiter Interoperabilität
		- Für eHealth Anwendungen muss es letztendlich egal sein, welche Software oder App und welchen Anbieter der Arzt, das Krankenhaus oder der Patient hat. Es muss so selbstverständlich klappen, wie telefonieren (= man kann jemanden problemlos anrufen, egal welches Endgerät oder Provider man hat).
	}

	2.2 Die Telematikinfrastruktur wurde nach dem Top-Down Ansatz entworfen. Beschreiben Sie diesen Ansatz und beschreiben Sie worin der Unterschied zum Bottom-Up Ansatz besteht?
	{
		Top-Down:
			- Der Gesetzgeber bestimmt
			- Die Industrie baut
			- Ärzte / Zahnärzte / Apotheker / Patienten usw. (müssen) machen

		Bottom-Up:
			- Einzelne Ärzte (Zahnärzte, Apotheker usw.) haben eine Idee
			- Ein Krankenhaus kommt dazu
			- Sie vernetzen sich, starten eine E-Health Anwendung
			- Oder ein Software-Hersteller bietet eine E-Health Anwendung an
			- Mehr Ärzte kommen hinzu, eine „Insel“ entsteht
			- Entscheidender Punkt: gelingt die Integration mit weiteren Inseln?
	}

	2.3 Nennen Sie jeweils 2 Vor- und Nachteile des Top-Down und des Bottom-Up Ansatzes?
	{
		Top-Down:
			+ Flächendeckende Vernetzung nach einem de-facto Standard
			+ Interoperabilität der so entwickelten E-Health Anwendungen
			+ Relevanter Markt für eHealth Hersteller (wirklich?)
			+ Kontrollierte Sicherheit
			- „Zwangsinfrastruktur“, erzeugt Widerstände
			- Die entwickelten Lösungen müssen nicht unbedingt die Besten sein (wir wissen es noch nicht)

		Bottom-Up:
			+ Freiwillige Lösungen, bessere Motivation für die Anwender
			+ "Besserer Wettbewerb ! nur gute Lösungen können sich etablieren"
			  ->theoretisch ja, klappt aber nicht gut... Wettbewerb wird oft durch technische Abschottung vieler Anbieter behindert
			- Insellösungen, in vielen Fällen inkompatibel untereinander
			- Häufig Abschottung von Anwendungen seitens der Hersteller
			- Keine flächendeckende Vernetzung
			- Sicherheitsanforderungen / -architektur?
			- Die Wahrscheinlichkeit für einen Patienten, dass alle seine Ärzte eine Anwendung unterstützen, ist klein
	}

	2.4 Was war der Lipobay Skandal (2001)? Warum war er für die Entwicklung der Telematikinfrastruktur bedeutend?
	{
		- Es gab nicht erkannte Wechselwirkungen zwischen zwei Medikamenten, da die behandelnde Ärzte nicht den Überblick besaßen, welche Medikamente ihre Patienten nahmen und wie die Wechselwirkungen zwischen den Medikamenten war
		- Daraufhin wurde 2003 mit dem GKV-Modernisierungsgesetz entgegengesteuert.
	}

	2.5 Nennen Sie 3 gesetzlich vorgeschriebene Anwendungen der Telematikinfrastruktur?
	{
		- Versichertenstammdatenmanagement (VSDM, Pflichtanwendung)
		- eRezept, Arbeitsunfähigkeitsbescheinigung (AU-Bescheinigung)
		- Notfalldatenmanagement (NFDM)
		- Kommunikation im Medizinwesen (KIM)
		- eMedikationsplan / Arzneimitteltherapiesicherheitsprüfung (AMTS)
		- ePatientenakte (Patientenfach, Patientenquittung, Impfpass, Mutterpass, weitere)
		- Organspendeerklärung
		- Hinweise über Organspendeerklärung, Patientenverfügung, Vorsorgevollmacht
	}

	2.6 Es gibt 2 Ansätze für den Aufbau einer TI. Welchen setzt die deutsche TI um?
	{
		Top-Down
	}

	2.7 Was sind Ziele der eGK und der TI?
	{
		Wirtschaftlichkeit, Qualität und Transparenz im Gesundheitswesen steigern, durch:
			- Vernetzung des deutschen Gesundheitswesens
			- Schaffung einer Infrastruktur / Plattform / „Datenautobahn des Gesundheitswesens“
			- Bausteine und Werkzeuge für bereits geplante und zukünftige E-Health Anwendungen
			- Sinnvolle und nutzbringende E-Health Anwendungen
	}

	2.8 Was ist unter Security-by-Design und Privacy-by-Default zu verstehen?
	{
		- Security by Design: Anwendungen sicher machen; Werkzeug für privacy by design; Sicherstellung der CIA Ziele;
		- Privacy by Design: Ich mach ein Systemdesign so, dass das System nur zulässige Daten erhebt, teilt. System berücksichtigt Datenschutzanforderungen schon in der Designphase einer Software. Bsp. Daten verschlüsselt erfassen; Nur notwendige Daten erfassen; Kontrolle über eigene Daten; Datenschutz(grundverordnung); Recht auf Löschung, Information von Anfang an ins Design implementieren
		- Privacy by Default: Das System arbeitet schon in der Grundeinstellung datenschutzkonform. Bsp. Nutzer muss nicht erst Verschlüsselung aktivieren.
		- Security by Default: Schon die Grundeinstellungen einer Software arbeiten sicher.

	}

	2.9 Wer sind die Akteure beim Aufbau/Bau der Telematik Infrastruktur?
	{
		- gematik GmbH (Berlin) gegründet im gesetzlichen Auftrag
		- Gesellschafter: Selbstverwaltung im Gesundheitswesen
			* Ärzte (Bundesärztekammer, Kassenärztliche Bundesvereinigung)
			* Zahnärzte (BZÄK, KZBV) • Apotheker (ABDA, DAV)
			* Krankenhäuser (Deutsche Krankenhausgesellschaft)
			* Krankenkassen (GKV-Spitzenverband)
			* BMG (Bundesministerium für Gesundheit)
	}

	2.10 Wieso wird eine landesweite Interoperabilität bei eHealth Infrastrukturen benötigt?
	{
		- Für eHealth Anwendungen muss es letztendlich egal sein, welche Software oder App und welchen Anbieter der Arzt, das Krankenhaus oder der Patient hat.
		- Es muss so selbstverständlich klappen, wie telefonieren (= man kann jemanden problemlos anrufen, egal welches Endgerät oder Provider man hat).
	}

	2.11 Was versteht man unter einer „Insel“ im Bottom-Up Ansatz bei der Entwicklung von eHealth Anwendungen?
	{
		- kleine Ideen werden an unterschiedlichen Orten umgesetzt
		- Lösungen sind inkompatibel (keine Interoparabilität)
		- Lösungen für individuelle Probleme für eine Gruppe von z.B. Ärzten
		- Verschiedene Regionen haben unterschiedliche Lösungen (= Insel)
	}

	2.12 Warum ist die Telematik Infrastruktur von besonderer Bedeutung und aus welchem Anlass wurde sie aufgebaut?
	{
		- Existieren von deutlichen Defiziten in der Vernetzung, Kommunikation und Zusammenarbeit im Gesundheitswesen, mithilfe der TI soll an dieser Stelle Abhilfe geschaffen werden
		- Anlass: schwerwiegende Nebenwirkungen bei Kombination von zwei Medikamenten  Lipobay-Skandal (beispielsweise haben verschiedene Ärzte je ein Medikament verschrieben, wussten aber jeweils nichts von der anderen Verschreibung
	}

	2.13 Beschreiben Sie 3 richtige Vorgehensweisen bei eHealth Projekten?
	{
		- Datenschutz und Informationssicherheit bereits im Design des Projektes und der Software berücksichtigen
		- Prozesse des Gesundheitswesens im Design berücksichtigen
		- Lösungen nach Bedarf des Gesundheitswesens entwickeln
	}

	2.14 Ist es sinnvoll eine landesweite eHealth Infrastruktur nach Bottom-Up Ansatz aufzubauen?
	{
		- Top-down:
			* Gesetzgeber bestimmt, andere müssen folgen
		- Bottom-up:
			* einzelne Ärzte oder Praxen haben Ideen, andere können mitmachen.

		-> Bottom-up Ansatz stellt die Medizin vor einige Herausforderungen, da sich sehr einfach Inseln bilden. Nur wenn diese zusammenschmelzen hat es am Ende ein nutzen für alle
		-> Top-Down macht auf jeden Fall mehr Sinn (Meiner Meinung nach)
	}

	3.1 Welche Vorteile hat die Entkopplung von Plattform und Anwendungen?
	{
		- Beliebige Anwendungen, die ins Architekturschema passen, können unterstützt werden, ohne Anpassung der Plattform
		- Gemeinsame Plattform-Dienste können unverändert von mehreren Anwendungen verwendet werden
	 	- Semantische Entkopplung (der Plattform ist egal, welche Informationen in der Anwendung ausgetauscht werden)
		- Entkopplung von Maßnahmen der Informationssicherheit
	}

	3.2 Erklären Sie die Funktionsweise der Card-to-Card (C2C) Authentifizierung.
	{
		- Direkte Authentisierung zwischen 2 Chipkarten
		- Mit Hilfe von Card Verifiable Certificates (CVC)
		- Damit können bei der eGK Daten freigegeben werden
			* z.B. Lesen/Schreiben Notfalldaten ohne PIN
		- Kryptographische Schlüssel können aktiviert werden
			* Zugriff mit Hilfe der eGK auf verschlüsselte Patientendaten nur bei Anwesenheit (also nach Freischaltung) eines HBA oder SMC
		- Rollenbasierte Autorisierung
			* z.B. Notfalldaten können nach Authentisierung eines eArztausweises, nicht aber eines eApothekerausweises auf die eGK geschrieben werden
		- „Sperrung“ der CVC über einen komplizierten Mechanismus, so dass ein gestohlener HBA (inkl. PIN) nicht mehr von eGKs (offline!) akzeptiert wird
			*„Zeit“-Information & Nachladbarkeit
	}

	3.3 Wie Funktioniert das „Zwei-Schlüssel-Prinzip“?
	{
		- Für jeden Datenzugriff müssen 2 Chipkarten zusammenarbeiten
			1. die eGK des Patienten
			2. der HBA oder Institutionskarte eines Leistungserbringers
		- aber nicht zwangsweise gleichzeitig:
		  -> Berechtigungskonzepte angedacht, so dass ein HBA/SMC mit Hilfe der eGK berechtigt wird, auch in Abwesenheit der eGK auf Daten zuzugreifen
		- Ausnahmen davon: Zugriff des Versicherten (i.d.R. über mobile Geräte)
	}

	3.4 Wie wird eine Nachvollziehbarkeit der Zugriffe auf die Patientendaten im TI-Umfeld sichergestellt?
	{
		-> Logging zur Datenschutzkontrolle
		- Alle Zugriffe auf Daten und möglichst alle Zugriffsversuche
		- Erst loggen, dann zugreifen
		- Log auf eGK (50 Einträge Ringspeicher)
		- und (falls online-Anwendung) Fachdienst online
	}

	3.5 Über welche Zone der TI-Architektur kann der Patient seine Daten einsehen?
	{
		Personal Zone -> (Zone in der TI Architektur)
	}

	3.6 Warum werden für Anwendungen in der TI Multireferenz-PINs eingesetzt?
	{
		Nach PIN-Eingabe soll nur eine Anwendung freigegeben werden und nicht global alle Anwendungen auf der Karte.
	}

	3.7 Erklären Sie die Eigenschaften von SOA in der TI-Architektur?
	{
		- Nachrichtenbasiert, SOAP (leider nicht REST), typischerweise mit Intermediär (Broker) für jede Anwendung
		- Mehrere Services (Fachdienste, Unterstützungsdienste)
	}

	3.8 Welche Zugangsmöglichkeiten gibt es in das geschlossene Netz der TI?
	{
		- Konnektor ← →  VPN-Konzentrator
		- Sicherheitsgateways/Proxies für externe (geschlossene) Netze und Anwendungen
		- Internet über Sicherheitsgateway (! sicheres surfen im Internet)
	}

	3.9 Welche Patientendaten sollen offline und dezentral auf der eGK gespeichert werden?
	{
		- Notfalldaten
		- Versichertenstammdaten
		- „Kommunikation im Medizinwesen“ (KIM, früher KOM-LE: Kommunikation der Leistungserbringer) → sichere E-Mail über S/MIME
		- Medikationsplan / AMTS (Arzneimitteltherapiesicherheitsprüfung)
	}

	3.10 Was wird als „Quantified Self“ bezeichnet?
	{
		- „The Quantified Self“ ist ein Netzwerk aus Anwendern und Anbietern von Methoden sowie Hard- und Softwarelösungen, mit deren Hilfe sie z. B. umwelt- und personenbezogene Daten aufzeichnen, analysieren und auswerten.
		- Es umfasst die Mittel und Methoden zur Erfassung von Daten aus allen Lebensbereichen.
		- “self knowledge through numbers”
	}

	3.15 Wie funktioniert die Ende-zu-Ende Verschlüsselung in der TI?
	{
		- Daten werden in einem Konnektor dezentral verschlüsselt und erst wieder in einem Konnektor oder in einer „Vertrauenswürdigen Ausführungsumgebung“ (VAU) für den Patientenzugang entschlüsselt
		- Schlüsselmanagement jedoch zentral, insb. Verschlüsselungsschlüssel
	}

	3.17 Was ist die zentrale Aufgabe des Konnektors im Ramen der TI?
	{
		Anbindung von Arzt- /Zahnarztpraxen, Krankenhäuser, Apotheken und weiterer Einrichtungen des Gesundheitswesens in das zentrale Netz der TI
	}

	3.18 Warum wurde der Top-Down Ansatz für den Ausbau der landesweiten eHealth Infrastruktur gewählt?
	{
		- Flächendeckende Vernetzung nach einem de-facto Standard
		- Interoperabilität der so entwickelten E-Health-Anwendungen
		- Relevanter Markt für E-Health Hersteller
		- Kontrollierte Sicherheit
	}

	3.19 Sind die Geräte/Anwendungen des Versicherten über Konnektoren verknüpft?
	{
		Nein. Sie sind an Fachdienste angebunden.
	}

	3.20 Welches Ziel hat der Aufbau der TI?
	{
		Interoperabilität
	}

	3.21 Nennen Sie 5 grundlegende Paradigmen der TI?
	{
		- TI als geschlossenes Netz/System
		- Alle Akteure/Geräte in der TI haben kryptografische Identität
		- Ende-zu-Ende Verschlüsselung
		- C2C Autentisierung
		- Offline vs. Online Speicherung von medizinischen Daten
		- Patientenindividuelle Verschlüsselung
		- Nachrichtenbasierte Kommunikation über SOAP
		- Zwei-Schlüssel-Prinzip
		- Logging zur Datenschutzkontrolle
		- Technische Autorisierung
		- Online Aktualisierung von Chipkarten durch CAMS
		- Für jede „Datei“ auf einer Chipkarte sind Rechte festgelegt
	}

	3.22 Was ist ein „Card Verifiable Certificate“ und inwiefern unterscheidet es sich von einem X 509?
	{
		-> CVC ist ein Public-Key-Zertifikat, das in einem besonders kompakten Format gespeichert wurde
		-> Gedacht für Kartenauthentifizierung
		- X509 ist deutlich größer
		- CVC ist minimiert
			* Kann auf der Karte liegen
			* Ermöglicht somit direkte Authentisierung zwischen zwei Chipkarten
	}

	3.23 Wie funktioniert der Zugriff auf die Daten der eGK ohne PIN?
	{
		- Über Card-to-Card Authentisierung (C2C)
			* Wenn ein Notfall passiert, Notfalldaten können nach Authentisierung eines eArztausweises auf die eGK geschrieben werden
		- Damit können bei der eGK Daten freigegeben werden
			* z.B. Lesen/Schreiben Notfalldaten ohne PIN
	}

	4.1 Welche Probleme sind bei dem mobilen Kartenterminal erkennbar?
	{
		- Zu kleiner Bildschirm
		- Mobiles Kartenterminal kann verloren gehen
		- Noch keine Nutzung von Tablets
	}

	4.2 Was versteht man unter dem Begriff des Primärsystems?
	{
		- Primärsysteme sind Systeme der Leistungserbringer (Praxisverwaltungssystem, AVS, KIS, usw.)
	}

	4.4 Nennen Sie 4 Eigenschaften von stationären Kartenterminals.
	{
		- Kommunikation mit Konnektor über Ethernet / TLS-abgesichert
		- Eigene kryptographische Identität (Pairing mit Konnektor)
		- 2 Einsteckplätze: HBA und eGK
		- PIN-Pad
		- Kleiner Bildschirm für Statusmeldungen (z.B. „Bitte geben Sie Ihre PIN zum Schreiben von Notfalldaten ein“)
		- Abgeschirmt, sicherheitszertifiziert
	}

	4.5 Nennen Sie 4 Merkmale der eGK.
	{
		- Werden von Krankenkassen ausgegeben, mit Foto
		- PIN (6stellig)
		- 96kB-128kB geschützter Speichers
		- sicherheitszertifizierter Chip und Betriebssystem (COS)
		- Nicht-auslesbare private Schlüssel und öffentliche Zertifikate (CVC und X.509-Zertifikate) zum Nachweis der Echtheit der Karte, technische Signatur/Authentisierung, Ver/Entschlüsselung von Daten
		- Root-CV-Zertifikate zur Validierung (Prüfung) der C2C-Authentisierung eines HBA/SMC oder CAMS
	}

	4.6 Nennen und erläutern Sie kurz 5 Kernfunktionen des Konnektors?
	{
		- Netzkonnektor: VPN-Device
		- Anwendungskonnektor
		- Anwendungsproxy / -server
		- Anbindung und Steuerung der Kartenterminals
		- Steuerung der Chipkarten
		- Anbieten von Basis-Diensten, wie Ver-/Entschlüsselung, Authentisierung
		- Signaturanwendungskomponente (Erzeugen und Prüfen von elektronischen Signaturen)
		-Vermitteln einer sicheren Internetverbindung
			-> (über Security Gateway in der TI)
		- Verbindung zum „Sicheren Netz der Kassenärztlichen Vereinigungen“ (SNK) und anderen mit der TI verbundenen Netzen und Anwendungen
	}

	4.7 Hat man automatisch Zugriff zur TI, wenn man ein neues Kartenterminal in einer Praxis ansteckt? Begründung?
	{
		Nein, da jedes Terminal eine kryptographische Identität hat und diese erst im Konnektor freigeschaltet werden muss.
	}

	4.8 Nennen Sie eine Möglichkeit wie der Konnektor eine sichere Verbindung zur TI aufbauen kann?
	{
		Über integriertes VPN mit IPsec
	}

	4.9 Zählen Sie 3 Anwendungen für Patienten in der Dänischen eHealth Infrastruktur auf?
	{
		- Online Terminvergabe, Kontaktinformationen, Warteliste, Qualitätsindikatoren, Bewertungen
		- Foren für Patienten
		- elektronische Patientenakte
		- eRezept
		- eMedikationsplan
		- Viedokonferenz, Telemonitoring
		- ...
	}

	4.10 Warum benötigen HBAs (Heilberufsausweise) 2 PINs?
	{
		Eine zur Signatur von Dokumenten und eine zur Authentifizierung
	}

	4.11 Weshalb gibt es ein stationäres und ein mobiles Kartenterminal und welche Eigenschaften hat das mobile Kartenterminal?
	{
		- stationär für Praxen/Krankenhäuser
		- mobiles Terminal für Haus- und Heimbesuche des Arztes

		Eigenschaften mobiles Kartenterminal:
			- Zwischenspeicher zur Speicherung mobil erfasster Daten
			- Daten werden für HBA des Arztes verschlüsselt gespeichert
			- (viel zu kleiner) Bildschirm gemäß Spec
			- Kleiner integrierter Anwendungskonnektor mit Fachlogik
			-> Es wurden bereits Konzepte eingereicht für die Nutzung von Tablets als Eingabe-/Ausgabegeräte inkl. Mobilfunkverbindung
		}

	4.12 Was versteht man unter „Health Literacy“? Wieso ist sie wichtig?
	{
		Grundlegendes Verständnis von Medizin und medizinischen Daten. Ist notwendig, dass der Patient die von ihm gespeicherten Daten verstehen kann.
	}

	4.13 Wieso ist eine Verbindung zwischen Primärsystem und Konnektor über TLS 1.2 nicht verpflichtend?
	{
		-> Praxisverwaltungssystem muss nicht TLS verwenden. TLS sollte aus sicherheitsgründen aber eingesetzt werden.
		"Die Gematik hat keine Zuständigkeit für den Bereich der Arztpraxen. Also für alles was außerhalb der Telematikinfrastruktur liegt. Also kann die Gematik auch nicht verpflichtend vorschreiben, dass die Primärsysteme eine TLS Verbindung aufbauen können müssen." - Georgios Raptis
	}

	4.14 Welche Anwendung wären für Ärzte in einer landesweiten eHealth Infrastruktur relevant? Nennen Sie mindestens 3.
	{
		z.B. eAkte, eArztbrief, eRezept
	}

	4.15 Nennen Sie 4 Eigenschaften von mobiler Kartenterminals?
	{

		- RBAC (Role-based Access Control)
		- ABAC (Attribute-based Access Control)
	}

	Z5.2 Welche Eigenschaften haben die Grundmodelle bzw. welche Unterschiede gibt es?
	{
		- DAC:
			* Zugriffsrechte basierend auf Subjekt S und Objekt O
			* Subjekt kann Zugriffsrechte eines Objektes festlegen
			* Z.B. Besitzer (Owner) eines Objektes legt Zugriffsrechte fest
			* Zugriffsrechte werden „pro Benutzer“ festgelegt -> keine Rollen
		- MAC:
			* Vergleich von Zugriffsebenen (Objekten) und Freigaben (Subjekte)
			* Systemweite Regeln legen Zugriffsrechte fest, können vom Subjekt nicht verändert werden
			* „Clearance“ von bestimmten Objekten und Subjekten
		- RBAC:
			* Subjekten werden Rollen zugewiesen
			* Rollen bestimmen Zugriffsrechte auf Objekte
		- ABAC
			* Subjekte, Objekte haben Attribute
			*Zugriffskontrollen-Entscheidung durch flexiblen Vergleich von Attributen
	}

	Z5.3 Welches Modell der Zugriffskontrolle implementiert ein Linux-Betriebssystem (ohne Erweiterungen, wie SELinux oder AppArmor)
	{
		- ACL (Access Control List)

			* Es kann Rollen und User-basiert festgelegt werden wer auf was Zugriff hat
	}

	Z10.8 Welche Ziele haben krankenhausinterne elektronische Patientenakten?

		- Umfassende und schnelle Information aller beteiligten Ärzte in einer Behandlungssituation -> Unterstützung der Zusammenarbeit
		- Effektive Suche nach bestimmten Informationen in der Akte
		- Nachvollziehbarkeit der Behandlung für rechtliche Zwecke
		- Unterstützung administrativer Anwendungen, wie z.B. Abrechnung

	Z10.9 Welche Verwendungszwecke haben krankenhausinterne elektronische Patientenakten?

		- Erfüllung gesetzlich vorgeschriebener nachgeordneter Zwecke wie Abrechnung, Meldepflichten, Qualitätsmanagement etc. (Sekundärer Zweck)
		- Forschung und Lehre, Gesundheitsberichtserstattung, Epidemiologie etc. (Tertiärer Zweck)
	}

	Z10.10 Welche Komponenten haben krankenhausinterne elektronische Patientenakten? Welche Funktionen erfüllen diese Komponenten?
	{

		- Metadatenspeicher, Document-Registry

	Z10.11 Welche Unterschiede bestehen i.d.R. in den Patientenakten zwischen Krankenhaus und Arztpraxis?

		- Arztpraxis:
			* Akte i.d.R. prozessorientiert
			* Laufende „Rolle“ mit Befunden, Notizen, Diagnosen, Maßnahmen, Dokumenten, usw. im Zeitverlauf
	}

	Z10.12 Beschreiben Sie verschiedene Strukturierungsgrade elektronischer Patientenakten.
	{
		- Hybride Akte -> unstrukturiert (häufig)
			* Elektronisch: Patienten-, Fall- und Abrechnungsdaten, ggf. Verweise zu Papierdokumenten
		- Elektrische Akte -> wenig strukturiert (häufig)
			* Patienten- und Abrechnungsdaten, Freitexteinträge in definierten Feldern, gescannte Papierdokumente oder formularmäßige elektronische Dokumente (z.B. PDF, Word)
		- Hochstrukturierte elektronische Akte
			* Kodierte Informationen anhand von kontrollierten Vokabularen/ Terminologien -> HL7 z.B.
	}

	Z10.13 Welchen Nachteil haben unstrukturierte elektronische Patientenakten bzgl. E-Health Anwendungen?
	{
		- Geringe Strukturierung der Daten -> Viel Freitext oder gar unveränderliche Dokumente (PDF), wenig kodierte Informationen
		- Semantische Interoperabilität schwierig
			* Bei Text: nur auf Feld-Ebene (z.B. Diagnosen)
			* Inhalt nicht maschinenauswertbar
	}

	Z12.1 Was sind IHE-Profile?
	{
		- Aus UseCases und relevanten Standards werden Leitfäden entwickelt, mit denen der Hersteller sein Produkt umsetzen und testen kann -> sog. IHE Profile
		- Integrationsprofile
		- Einschränkung eines Standards durch Profilierung
	}

	Z12.2 Mit welcher Motivation wurden IHE Profile entwickelt?
	{
		- HL7, DICOM, usw.  haben sehr viele (zu viele) Optionen
		- Es ist sehr schwer, praktisch unmöglich für einen Hersteller, sämtliche Optionen eines Standards zu unterstützen
		  -> Im Ergebnis keine Interoperabilität
	}

	Z12.3 Was ist das Ziel von IHE-Profilen?
	{
		- Nutzt vorhandene Standards, um im Kontext eines Geschäftsprozesses Interoperabilität zu erzielen
		- Ziel ist immer die interoperable Kommunikation medizinischer Informationen im Kontext klinischer Prozesse
	}

	Z12.4 Welche Standards setzen IHE-Profile ein?
	{
		!!!Nicht sicher!!!
		- HL7, DICOM, ...
	}

	Z12.5 Was sind Transaktionen im Kontext der IHE-Profile?
	{
		- So etwas wie eine Funktion -> Es wird für eine Transaktion genau beschrieben aus welchen Schritten und über welche Schnittstellen Informationen ausgetauscht werden
		- Z.B. „PatientRegistration“ oder Anforderung bestimmter Daten
		- Transaktionen werden zwischen Akteuren definiert
		- Austausch von Dokumenten, Suchanfragen, usw.
	}

	Z12.6 Was bedeutet es, dass IHE-Profile Kommunikationsprotokolle im Kontext klinischer Prozesse definieren? Was ist damit gemeint?
	{
		!!!Nicht sicher!!!
		- Innerhalb einer Domain wird für die Kommunikation ein Protokoll festgelegt
		- Dadurch wird Interoperabilität ermöglicht
	}

	Z12.7 Was ist eine XDS Affinity Domain?
	{
		- Gruppe von Einrichtungen im Gesundheitswesen, die kooperieren und Gesundheitsdaten austauschen
		- mit gemeinsamen Policies
	}

	Z12.8 Wofür wird IHE-XDS.b eingesetzt?
	{
		- (Cross Enerprise Document Sharing)
		- Registrierung / Speicherung / Abruf von Submission-Sets („Ordner“ mit mehreren Dokumenten)
		- XDS.b verarbeitet beliebige Inhalte -> Definiert also nicht, was transportiert werden darf
	}

	Z12.9 Welche Funktion hat die Registry, welche das Repository in IHE-XDS.b?
	{
		- Source:
			* Datenquelle -> Liefert Dokumente und Metadaten an Repository
		- Repository:
			* Speichert Dokumente
			* Läuft i.d.R. lokal (z.B. im Krankenhaus), kann aber auch woanders sein
			* Registriert Dokumente an Registry
			* Liefert Dokumente, die abgerufen werden, anhand deren Identifier
		- Registry:
			* Speichert Metadaten der Dokumente
			* Unterstützt Suchabfragen
			* Nutzt dafür die Patientenidentifikation
		- Consumer:
			* Sucht nach Dokumenten im Registry
			* Ruft dann gefundene Dokumente ab
		- Patient Identity Source:
			* Liefert Informationen zur Patientenidentifikation, insbesondere eine eindeutige Patienten-ID
	}

	Z12.10 Welche Funktionen übernimmt das IHE-ATNA Profil?
	{
		- (Audit Train and Node Authentification)
		- Authentifizierung / Verschlüsselung der Kommunikation zwischen Maschinen
		- Zentrale Protokollierung (Auditing) aller Aktionen -> Wer hat was gemacht
	}

	Z12.11 Wofür wird das IHE-Profil IHE-MHD verwendet?
	{
		- (Mobile access to Health Documents)
		- Definiert ein http Interface für eine Document Sharing Umgebung
		- V.a. gedacht für Applikationen die lieber über HTTP als XDS zugreifen wollen
	}

	Z12.12 Wofür wird das IHE-Profil IHE-APPC verwendet?
	{
		- (Advanced Patient Privacy Consents)
		- Contentprofil, mit dem patienten-spezifische Einwilligungen erfasst, verwaltet und zwischen verschiedenen Systemen und Organisationen ausgetauscht werden können
	}

		Es muss so selbstverständlich klappen, wie telefonieren (= man kann jemanden
		problemlos anrufen, egal welches Endgerät oder Provider man hat).
	}

	F40: Welche 2 Möglichkeiten gibt es eine landesweite E-health Infrastruktur aufzubauen? Nenne!
	{
		1. Top-Down
		2. Bottom-Up
	}

	F41: Was versteht man unter dem "Top-Down" Ansatz?
	{
		- Gesetzgeber bestimmt
		- Industrie baut
		- Ärzte/ Apothekter/ Patienten (müssen) machen/ benutzen
	}

	F42: Was versteht man unter dem "Bottom-Up" Ansatz?
	{
		- Einzelne Ärzte/ Apotheker/ usw. haben eine Idee
		- Ein Krankenhaus kommt dazu
		- Sie vernetzen sich und starten eine E-Health Anwendung
		- Oder Software-hersteller bietet eine Anwendung an
		- Mehr und mehr Ärzte kommen hinzu

			-> Eine Insel entsteht
			-> Enscheidender Punkt: Gelingt die Integration mit weiteren Inseln
	}

	F43: Nennen Sie Vor- und Nachteile des "Top-Down" Ansatzes.
	{
		+ Flächendeckende Vernetzung nach de-facto Standard
		+ Interoperabilität der entwickelten Anwendung
		+ Kontrollierte Sicherheit
		+ Relevanter Markt für E-Health Hersteller -> Hat sich noch nicht bewahrheitet

		- "Zwangsinfrastruktur" -> Erzeugt Widerstände
		- Die entwickelten Lösungen müssen nicht zwingend die Besten sein
	}

	F44: Wie wird eine kontrollierte Sicherheit in der TI gewährleistet?
	{
		- Vorgaben und Kontrolle durch BSI
		- Prüfung durch die Gesellschafter der Gematik
		- Definiertes Sicherheitsniveau
			-> Spezifikationen werden veröffentlicht -> Transparenz
			-> Security by Design
			-> Privacy by Design
			-> Also nicht durch Geheimhaltung, sondern durch bloßes Design
	}

	F45: Nennen Sie Vor- und Nachteile des "Bottom-Up" Ansatzes.
	{
		+ Freiwillige Lösungen -> Bessere Motivation
		+ Theoretisch besserer Wettbewerb
			-> Technische Abschottung verhindert das aber oft

		- Insellösungen oft inkompatibel
		- Technische Abschottung von Anwendungen
		- Keine flächendeckende Vernetzung
		- Fehlende Sicherheitsanforderungen/ -architektur?
			-> Da teilweise von kleinen Firmen hergestellt (Sicherheit kostet)
		- Wahrscheinlichkeit für einen patienten, dass sein Arzt eine Anwendung unterstützt, ist sehr klein
			-> Zerstückelung des Marktes
	}

	F46: Warum gab es in einem Land noch nie eine flächendeckende, interoperable, sichere, landesweite eHealth Infrastruktur nach dem Bottom-Up Ansatz?
	{
			- Sicherheit und Datenschutz kosten
				-> macht keiner freiwillig
			- Interoperabilität läuft gegen viele Geschäftsinteressen
				-> Verteidigung von Marktanteilen
			- Viele Ärzte sehen keinen Grund sich zu vernetzen
				-> Patientenbindung durch Abschottung
				-> Angst vor Transparenz

			-> Es braucht also Regulierung durch den Staat (!Sogar in den USA so)
	}

	F47: Welche Anwendungen sind in der TI gesetzlich festgeschrieben?
	{
		- VSDM
		- eRezept, AU-Bescheinigung
		- NFDM
		- KIM (Kommunikation im Medizinwesen)
		- eMedikationsplan & AMTS
		- ePatientenakte
			-> Patientenfach, Patientenquittung, Impfpass, Mutterpass, ...
		- Organspendeerklärung
		- Hinweise über Organspende, Patientenverfügung, Vorsorgevollmacht
	}

	F48: Aus welchen grundlegenden Architekturbestandteilen besteht die TI?
	{
		- Plattform
		- Anwendung
	}

	F49: Was versteht man und der "Entkopplung von Plattform und Anwendungen" in der TI?
	{
		- Beliebige Anwendungen, die ins Architekturschema passen, können unterstützt werden, ohne Anpassung der Plattform
			-> aber z.B. keine nicht nichrichten-basierte Kommunikation wie Videotelefonie oder  Anwednungen die eine Rest Schnittstelle nutzen
			-> dann müsste man Netzwerkschnittstelle nutzen
		- Gemeinsame Plattform-Dienste können unverändert von mehreren Anwendungen verwendet werden
			-> z.B. Verschlüsselungsdienste
		- Semantische Entkopplung (der Plattform ist egal, welche Informationen in der Anwendung ausgetauscht werden
			-> Kontent ist egal
		- Entkopplung von Maßnahmen der Informationssicherheit
			-> Anwendung muss die Dienste zur Info.Sicherheit nich nutzen, sie kann
			-> Plattfrom schreibt nicht vor, was zu nutzen ist
			-> Das Gesetz vllt schon
	}

	F50: Beschreiben Sie die TI als Plattform.
	{
		- Service-orientierte Architektur (SOA)
			* nachrichtenbasiert
			* typischerweise Kommunikation aller Anwednungen über Intermediär (Broker)
			* mehrere Services (Fachdienste, Unterstützungsdienste)
		- Zentrales Netz
			* VPN, MPLS-basiertes Backbone
			* Sicherheitsgateways für Anbindung mit anderen Netzen
		- Anbindung per "Konnektor"
			-> Hardwaregerät
	}

	F51: Welche Zonen gibt es in der TI-Netzwerktopologie?
	{
		- Consumer Zone
			-> Rechner der Arztpraxis
		- Secure Consumer Zone
			-> Anwendung auf dem Rechner in der Arztpraxis
			-> Konnektor verbindet diese Zone mit der dezentralen TI-Plattform Zone
		- TI-Plattform Zone dezentral
			-> Konnektor
			-> kartenterminal
			-> Chipkarten
		- TI-Plattform Zone zentral
			-> zentrales Netz der TI
		- Personal Zone
			-> Frontends des Versicherten auf Geräten des versicherten
			-> z.B. E-Health App
		- Existing Application Zone
			-> z.B. Datenbanken der Krankenkassen
			-> oder andere Anwendungen externer Netze
		- Provider Zone
	}

	F52: Zählen Sie die Paradigmen der TI auf. (Vllt sehr dumme Frage)
	{
		- TI ist ein geschlossenes Netz
		- Alle Akteure und Geräte in der TI haben eine kryptographische Identität
		- Ende-zu-Ende verschlüsselte Kommunikation
		- Kommunikation is nachrichten-basiert (SOAP)
		- Patientenindividuelle Verschlüsslung
		- Online und Offline Speicherung von medizinischen Daten
		- Card-to-Cart Authentifizierung
		- Zwei Schlüssel Prinzip
		- Logging zur Datenschutzkontrolle
		- Technische Autorisierung (gesetzlich festgelegt)
		- Für jede Datei auf einer Chipkarte sind Rechte festgelegt
		- Online Aktualisierung von Chipkarten durch CAMS
	}

	F53: Inwiefern ist die TI ein geschlossenes Netz?
	{
		- Kein unkontrollierter Zugang zum Internet
		- Zugang über Konnektoren
		- Außenzugang der TI über:
			* VPN-Konzentratoren
			* Sicherheitsgateways/ proxies für externe Netze/ Anwendungen
			* Internet über Sicherheitsgateways
	}

	F54: Wie ist die kryptographische Identität in der TI umgesetzt?
	{
		- i.d.R. in Form einer Chipkarte
		- Gegenseitige krypt. Authentifizierung aller Akteure
		- Sperrung einer krypt. Identität ist jederzeit möglich
	}

	F55: Wie wird verhindert, dass Daten durch MITM abgegeriffen werden können?
	{
		- Durch Ende-zu-Ende Verschlüsselung
		- Daten werden vor Übertragung im Konnektor dezentral verschlüsselt
		- Entschlüsselung im Empfängerkonnektor oder einer vertrauenswürdigen Ausführungsumgebung (VAU)
		- Schlüsselmanagement jedoch zentral, insb. Verschlüsselungsdienste
	}

	F56: Was ist Card-to-Cart (C2C) Authentifizierung?
	{
		- Direkte Authentisierung zwischen 2 Chipkarten
			-> mithilfe von Card Verifiable Certificates (CVC)
		- Dadurch können bei der eGK Daten freigegeben werden können
			-> z.B. für Lesen von Notfalldaten ohne PIN
		- HBA oder SMC werden benötigt, um kryptographische Schlüssel zu aktivieren
			-> Rollenbasierte Autorisierung
			-> Daten/ Zugriff je nach Rolle
		- Sperrung der CVCs auch für offline möglich
			-> z.B. gestohlener HBA ungültig machen
	}

	F57: Welche Anwendungen nutzen online-Speicherung und welche offline?
	{
		online:
			- ePatientenakte
			- eRezept, AU-Bescheinigung
		offline (keine langfristige zentrale Speicherung, dezentral auf der eGK, oder nur Datenübertragung):
			- Notfalldaten
			- KIM
			- eMedikationsplan, AMTS
		online & offline:
			- Versichertenstammdaten
	}

	F58: Was ist das WZei-Schlüssel-Prinzip?
	{
		- Für jeden Datenzugriff müssen mindestens 2 Chipkarten zusammenarbeiten
			1. eGK des Patienten
			2. HBA oder Institutionskarte eines Leistungserbringers
			-> Über Kartenterminal
			-> Nicht zwangsweise gleichzeitig
				-> Auch in Abwesenheit des Patienten berechtigt auf Daten zuzugreifen

		- Ausnahme: Zugriff des Versicherten über mobile Geräte auf eigene Daten auch ohne Arzt
	}

	F59: Wie wird Datenschutzkontrolle in der Ti umgesetzt?
	{
		- Logging aller Zugriffe auf Daten und möglichst alle Zugriffsversuche
		- Erst logen, dann zugreifen
		- Laug auf eGK -> (50 Einträge Ringspeicher)
		- Online Fachdienst regelt, falls Online-Anwendung
	}

	F60: Was gitb es über die technische Autorisierung zu wissen?
	{
		- vorgeschriebene PIN-Eingabe für alle freiwilligen Anwendungen
			-> manche brauchen sogar zusätzlich C2C-Authentifizierung
		- "Virtuelle" PINs für jede Anwendung
			-> Multireferenz-PINs:
				* Es gibt nur eine reale Karten-PIN der eGK
					-> zeigt auf PIN.CH, hat aber eigene Security Status
				* nach Pineingabe wird nur eine Anwendung freigegeben und nicht global alle
		- Keien PIN-Eingabe für Notfalldaten
	}

	F61: Welche Funktionen hat ein Konnektor?
	{
		- Sichere Netzanbindung in die Telematik-Infrastruktur
		- Anwendungsproxy / -server
		- Anbindung und Steuerung der Kartenterminals
		- Anwendungskonnektor
		- Steuerung der Chipkarten
		- Anbieten von Basisdiensten(Authentifizierung, Ver-/Entschlüsselung)
		- Signaturanwendungskomponente
		- Vermitteln einer sicheren Internetverbindung -> Sicherheitsgateway
		- Verbindung zu mit der TI verbundenen Netzwerken (z.B. SNK)
	}

	F62: Inwiefern ist der Konnektor modular?
	{
		- Basisdienste und Anwendungen werden als Module angeboten
		- Module können prinzipiell ersetzt werden
			-> Möglichkeit Konnektor zu erweitern
		- Jede Anwendung hat im Konnektor sein eigenes Fachmodul
			-> implementiert Fachlogik der Anwendung
		-> SOAP Aufruf an Konnektor -> Modul nutzt Basisdienste
		- Idee: Basisdienste werden verwendet um Module zu "scripten"
			-> mit anwendungsspez. Parametern und Logik
	}

	F63: Wird die Modularität des Konnectors wirklich so umgesetzt wioe geplant?
	{
		- Noch nicht
		- oft monolithische Struktur und hardcoded
		 -> Aber später sollen sie nachträglich modularisiert werden
		 -> Vllt wird es aber auch ein Software-Konnektor wie in USA
	}

	F64: Was ist die eGK? Nennen Sie ein paar Eigenschaften.
	{
		- elektr. Gesundheitskarte
		- mit Foto
		- wird von Krankenkassen ausgegeben
		- PIN(6-stellig)
		- 96-128kB geschützter Speicher
		- sicherheitszertifizierter Chip und Betreibssystem
		- verfügt über:
			* nicht-auslesbare private Schlüssel
			* öffentliche Zertifikate(CVC und X.509-Zertifikate) zun Nachweis der Echtheit der Karte
			* technische Signatur/Authentifizierung
			* Ver- und Entschlüsselung von Daten
			* Root-CV-Zertifikate zur Validierung der C2C-Authentifizierung eines HBA/SMC oder CAMS
			* NFC-Schnittstelle
			* qualifizierte elektr. Signatur
	}

	F65: Was sind HBA und SMC? Und wofür werden Sie verwendet?
	{
		- Institutionskarten
		- HBA -> Heilberufsausweis
		- SMC -> Secure Module Card
		- Signaturkarte
		- Authentifzierung und Autorisierung in TI-Netzwerk oder gegenüber anderen Ausweisen

	}

	F66: Eigenschaften von HBA und SMC?
	{
		- 2 PINs (6-stellig) -> für Signatur und alle anderen Funktionen
			-> 1 PUK (8-stellig)
		- sicherheitszertifizierter Chip und Betriebssystem
		- zusätzliche NFC/RFID Schnittstelle
		- X.509 Public Key Zertifikate inkl. Berufsattribut und Schlüssel für
			* Qualifizierte elektronische Signatur
			* Authentisierung (z.B. im Rahmen einer TLS-Verbindung)
			* Ver-/Entschlüsselung von Daten (inzwischen nur HBA, außerhalb der TI)
	}

	F67: Beschreiben Sie ein stationäres Kartenterminal.
	{
		- Kommunikation mit Konnektor über Ethernet/ TLS-abgesichert
		- Eigene kryptographische Identität (Pairing mit Konnektor)
		- 2 Einsteckplätze für HBA und eGK
		- PIN-Pad
		- Kleiner Bildschirm für Statusmeldungen
		- Abgeschirmt, sicherheitszertifiziert
	}

	F67: Beschreiben Sie ein mobiles Kartenterminal.
	{
		- Zwischenspeicher zur Speicherung mobil erfasster Daten
		- Daten werden für HBA des Arztes verschlüsselt gespeichert
		- recht kleiner Bildschirm
		- kleiner integrierter Anwendungskonnektor mit Fachlogik
		-> Idee für Tablets als neues Konzept
	}

	F68: Was sind "Primärsysteme? Nenne auch Beispiele."
	{
		- Systeme der Leistungserbringer
		- haben keine direkte Verbindung zur TI
			-> aber nachrichten-basierte Kommunikation über Konnektor
		- Keine direkte Steuerung der Fachdienste der TI, sondern nur über Module des Konnektors
			-> steuern dann die eigentlichen Fachdienste
		- Beispiele:
			* PVS, KIS
	}

	F69: Erklären und unterscheiden Sie kurz die Begriffe Authentifizierung, Autorisierung und Identifikation.

		- Autorisierung: Einräumung von speziellen Rechten basierend auf einer Identität
		- Identifikation: Behauptung einer Identität

	F70: Welche grundsätzlichen Methoden der Authentifizierung gitb es?

		- Wissen
		- Besitz
		- körperliches Merkmal -> ist auch irgendwie Besitz

	F70: Nenne Beispiele für Arten sich zu authentifizieren?

		- Biometrische Daten, z.B. Fingerabruck, Face-ID
		- Chipkarte
	}

	F71: Was ist Opt-Out? Erkläre anhand eines Beispiels.

	F72: In der Schweiz gibt es ein Multilevel-Security-Zugriffsmodel. Was ist das?
	{
		- Patient vergibt Vertraulichkeitssstufe an Dokumente -> Classification
		- Patient vergibt Zugriffsstufe an "Gesundheitsfachpersonal" -> Clearance
		- Zugriff, wenn Classification <= Clearance
		- Notfallzugriff ohne explizite Rechtevergabe
			-> kann aber abgeschalten werden
	}

	F73: Wie sollte man in einer E-healthanwendung am besten Authentifizierung umsetzen?
	{
		- Selber machen in der Anwendung -> nicht zu empfehlen
		- Integration eines identity Providers -> z.B. openID Connect
	}

	F74: Worin unterscheidet sich ein Challenge-Response-Verfahren zu Passwort, TAn, usw.?
	{
		- Wissensnacheis zur Authentifizierung wird bei CR-Verfahren nicht direkt an den Server übertragen
		- Dadurch kann das Passwort nicht einfach durch MITM abgefangen und gestohlen werden
	}

	F75: Wie genau funktioniert ein Challenge-Response-Verfahren?
	{
		- Server schickt "Challenge" an Client
		- Erstellung einer elektr. Signatur auf die Challenge mit Hilfe eines krypt. privaten Schlüssels (von z.B. der Chipkarte)
			-> Ergebnis (Response) an Server
			-> Schlüssel wird nicht nich direkt übertragen
		- Ergebnis wird serverseitig mittels elektr. Zert. überprüft
			-> z.B. X.509-Zertifkat wenn Chipkarte
	}

	F76: Welche Arten gibt es eine Chipkarte zu aktivieren?
	{
		- PIN
		- kryptographische Authentifizierung -> C2C mit HBA/SMC
		- biometrische Aktivierung -> kommt in Zukunft irgendwann

		-> Setzen von Security Condition, um bestimmte Informationsobjekte (Dateien, Schlüssel) der Karte lesen zu können
	}

	F77: Unterschied zwischen klassischer und kryptographischer Zugriffkontrolle?
	{
		- Software kann leicht gehackt werden
		- Mathematik um einiges sicherer -> sehr gute Kryptographieverfahren
		- Kryptographie als zusätzlicher Schutz -> klassiche Zugriffskontrolle wird zusätzlich zu Kryptographie verwendet
	}

	F78: Was ist Hybridverschlüsselung?
	{
		- Symm. Schlüssel wird verschlüsselt mit asymm. öffentlichem Schlüssel aller Berechtigten
			-> Am Ende so viele asymm. verschlüsselte symm. Schlüssel wie Berechtigte in ACL
			-> z.B. ein Eintrag mit Schlüssel für Arzt, der Zugriff hat und ein Eintrag für den Patienten
	}

	F79: Wie funktioniert das Konzept kryptographisch unterstützter Zugriffskontrolle auf Dokumentenebene?
	{
		- Policy auf Ebene von Dokumentenklassen (alle Dokumente einer Anwendung)
		- Patient kann aber Rechte für einzelne Dokumente individuell festlege

		Es gibt:
			- Objekt-Tickets: mit Informationsobjekt assoziierte ACL
				-> enthält alle Zugriffsberechtigungen für das Info-Objekt und die Kryptoschlüssel
			-Service-Tickets: "Template" einer ACL für eine Info-Objekt-Klasse
				-> Bei jedem neuen Info-Objekt dieser Klasse werden Objekt-Tickets nach dem Muster des Service-Tickets erstellt

		Zugriff auf Informationen:
			- Authentifizierung
			- Rechtemanagement entscheidet ob Zugriff ok
			- Datenübertragung
			- Entschlüsselung des Dokumentenschlüssels aus Ticket mit dem provaten Schlüssel des Zugreifenden
			- Entschlüsselung des Dokuments mit gerade gewonnenem symm. Dokumentenschlüssel
			-> Dokumente lesbar

		Rechtemanagement: Bsp:  Dauer-Berechtigung eines Arztes zum Zugriff auf bereits vorhandene PflegeakteDokumente des Patienten
			- Erstellung eines Service-Tickets: enthält ID und öffentlichen Schlüssel des Arztes
			- Signatur des Service-Tickets durch eGK des Patienten
			- Bei jedem neuen Pflegeakte-Dokument wird anhand des Service-Tickets im zugehörigen Objekt-Ticket
				* die ID des berechtigten Arztes aufgenommen
				* der Dokumentenschlüssel mit dem öffentlichen Schlüssel des Arztes verschlüsselt und im Ticket aufgenomme
			- Alle Tickets vom Typ „Pflegeakte-Dokument“ des Patienten werden (nach Authentifizierung der eGK) lokal geholt
			- Die verschlüsselten Dokumentenschlüssel werden mit dem privaten eGK-Schlüssel entschlüsselt
			- wieder zusätzlich mit dem öffentlichen Schlüssel des Arztes verschlüsselt
			- die Tickets werden mit neuen Einträgen für den berechtigten Arzt ergänzt
			-> Arzt hat nun Berechtigung und Eintrag im ACL
	}

	F70: Wie funktioniert das Konzept kryptographisch unterstützter Zugriffskontrolle auf Anwendungsebene?
	{


		- Rechte für alle Dokumente einer Anwendung (z.B. elektronische Patientenakte)
		- Patient kann Arzt/Krankenhaus für die gesamte Akte autorisieren
			-> Alles oder nichts Prinzip

		Anwendungsschlüssel:
			- Dokument wird mit einem individuellen symmetrischen Schlüssel verschlüsselt
			- Alle symmetrischen Dokumentenschlüssel der Anwendung werden mit einem Anwendungsschlüssel (pro Patient) symmetrisch verschlüsselt
			- Der Anwendungsschlüssel wird mit den jeweiligen asymmetrischen öffentlichen Schlüssel der Berechtigten verschlüsselt
				* Zugriffsmuster: priv. Schlüssel z.B. auf HBA kann Anwendungsschlüssel entschlüsseln. Mit Anwendungsschlüssel können Dokumentenschlüssel entschlüsselt werden. Damit können die Dokumente entschlüsselt werden
				* Berechtigter erhält dann Zugriff auf alle Dokumente der Anwendun
	}

	F81: Was sind Nachteile der kryptographisch unterstützten Zugriffskontrolle?
	{
		- Kompliziertes Rechtemanagement
			-> Viele Krypto Opperationen
			-> Rechteanpassung ist viel mehr als nur Anpassung eines Eintrags im ACL
		- Kompliziertes Schlüsselmanagement
			-> Wenn Schlüssel weg, Daten unzugänglich
				-> Datenerhalt-Konzepte notwendig
	}

	F82: Warum braucht man Datenerhalt-Konzepte?
	{
		- Erforderlich ist eine „Umschlüsselung“ aller zentral gespeicherten Gesundheitsdaten eines Versicherten, wenn die eGK ausgetauscht wird
		- Muss auch gehen, wenn eGk verloren/gestohlen
	}

	F83: Welche Anforderungen gibt es an ein Datenerhalt-Konzept?
	{
		- Keine Generalschlüssel
		- Keine Speicherung des privaten Schlüssels
		- Aktive beteiligung des Versicherten
		- Muss auch nach Kartenausgabe noch implementierbar sein
	}

	F84: Welche administrativen Zugänge zum Gesundheitssystem gibt es?
	{
		- gesetzlichen Krankenversicherungen (Krankenkassen)
			-> Arbeitnehmer in abhängiger B<eschäftigung oder Rentner
		- private Krankenversicherungen
			-> Gewerbetreibende, Unternehmer, Beamte, ...
	}

	F85: Welcher Dokumente für den Berechtigungsnachweis gibt es?
	{
		- Krankenschein
		- Krankenversichertenkarte (KVK)
		- elektronische GEsundheitskarte (eGK)
	}

	F86: Nenne Verbesserungen der eGK gegenüber der KVK.

	F87: Wie funktioniert der Versichertennachweis auf der eGK offline?

	F87: Wie funktioniert der Versichertennachweis auf der eGK online?

			-> Nutzen des Update Flag Service (UFS)

			- nein: elektronische Quittung wird von UFS ausgestellt
				-> VSD aus eGK ä Quittung werden für Abrechnung verwendet
			- ja: Aufbau Verbindung zu VSD-Dienst
				-> Datensatz auf Chipkarte wird online aktualisert
				-> elektronische Quittung wird vom VSDM ausgestellt und für Abrechnung genutzt
	}

	F88: Wofür wird der Versucherungsnachweis hauptsächlich verwendet?
	{
		- Für die Abrechnung des Arztes
	}

	F89: Wie läuft der Prozess einer Datenänderung auf der Chipkarte ab?
	{
		- ging früher mit KVK nicht und mit offline eGK
		- geht nur online mit eGK über Online VSDM:
			- Versicherte melden neue Anschrift an Krankenkasse
			- Krankenkasse pflegt Daten in VSDM-Datenbank ein
			- Update Flag des UFS wird gesetzt
			- Bei nächstem Kontakte der eGK mit TI: Online Aktualisierung über VSDM Dienst
	}

	F90: Wie unterscheidet sich das VSDM von Deutschland zu Österreich?
	{
		- VSD sind auf der österreichischen eCard nicht auf der Karte gespeichert
	}

	F91: Inwiefern kommuniziert die Arztpraxis mit den Krankenkassen nur asynchron und warum?
	{
		- Kommunikation läuft über eGK als Intermediär/Mittelsmann
		- Dadurch wird Identität des arztes anonymisiert
			-> Sonst könnten Krankenkassen Bahandlung beeinflussen
	}

	F92: Wo wird das eRezept/ elektronische Verordnung in DE bereits eingesetzt?
	{
		- In Closed-Loop Prozessen intern in Krankenhäusern -> eigene Implementierungen
	}

	F93: Welchen Nutzen hat eine elektronische Verordnung / eRezept?
	{
		- Lesbarkeit erhöhen -> schlechte Handschrift (Wenn Rezept manuell geschrieben)
		- Fehler vermeiden
			- Medienbruch vermeiden
		- elektronische Plausibilitätsprüfung
		- Kosteneinsparung durch eRezept
		- Kontrolle und Transparenz
	}

	F94: Warum wurde eRezept in DE noch nicht eingeführt?
	{
		- zu früher test von unreifem Prozess -> Öffentlichkeit wurde gebrandmarkt
		- Prozess der Papier-verordnung ist (zu) gut
			-> Kein dringlicher Bedarf
			-> in DE eRezept mit Papier als Datenträger
	}

	F95: Welche Umsetzung für eRezept fallen dir ein?
	{
		- Offline-Anwendung -> Chaipkarte z.B. als Datenträger
		- Online-Anwendung -> WebService-Lösung (SOAP/REST) -> Kommunikation über PVS
		- Hybrides Konzept -> Offline und Online Möglichkeit
			-> Speichern auf Chipkarte möglich oder Verwendung von Online-Dienst und speichern von Pointer auf Karte
	}


	F96: Nenne Sie einen konzeptionellen Vor- und Nachteil des Hybriden eRezept Konzeptes von 2008.
	{
		+ Hoher Datenschutz
		- Hohe Komplexität
		- Es können damit nicht alle etablierten Prozesse abgebildet werden
			-> z.B. Einlösung von Rezepten durch Verwandte, Diskretversand durch Online-Apotheken
	}

	F97: Welche Probeleme kommen beim Offline eRezept auf im Vergleich zur Online Lösung?
	{
		- Physische Präsenz beim Arzt erforderlich
			-> Doof für Wiederholungsrezepte
			-> Doof für Online-Apotheken
		-  Authentizität, Integrität und Verbindlichkeit des Datensatzes schwieriger zu gewährleisten
			-> Kein allgemeine Sicherung durch Zugangsschutz des Fachdienste
			-> elektronische Signatur nötig
		- Es muss sichergestellt werden, dass ein Rezept nicht zweimal verwendet wird
			-> Online-Prüfung von Duplikaten
			-> Mindestens das muss Online geschehen
	}

	F98: Was ist eine Patientenquittung?
	{
		- Wenn man zum Arzt geht, wird man untersucht
		- Arzt muss diese Behandlung sehr genau nach Katalog festhalten
		- diese Informationen stehen dann auch in den Abrechnungsinformationen
			-> Patient bekommt diese nur auf Anforderung
	}

	F99: Was versteht man unter "Upcoding"?
	{
		Kontext: Nutzung von Abrechnungsdaten für medizinisches Zwecke
		-> Arzt macht Patienten auf Papier kränker als sie sind
		-> sodass er mehr Geld bekommt
	}
